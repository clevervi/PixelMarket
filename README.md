# Ancestral Heartbeat (Latido Ancestral)
Multi-vendor e-commerce platform built with **Next.js 16 (App Router)**, **TypeScript**, and **Supabase** (PostgreSQL + Auth).

This README documents the current state of the application (what already exists in this repo today), its architecture, how to run it locally, and how to deploy it to Vercel. It also includes a forward-looking roadmap for upcoming iterations.

## Product overview
**Ancestral Heartbeat / Latido Ancestral** is an online store focused on Colombian handicrafts. The project is designed as a marketplace: multiple vendors can manage catalog and orders, while an admin oversees the platform.

## Current scope (what is implemented today)
### Storefront (public)
- Public pages: home, shop, collections, products, inspiration, essence, contact, terms, privacy.
- Product detail with gallery, recommendations, and review/Q&A sections.
- Cart and wishlist on the frontend.

### Checkout and orders
- Checkout builds the order payload and sends it to the backend.
- Order creation in PostgreSQL via API (`pedidos`, `detalle_pedido`, `direcciones`).
- Coupon validation and usage tracking (`cupones`, `cupones_usuarios`).

### Authentication
- Supabase Auth (email/password).
- Internal `auth_token` cookie signed with `AUTH_SECRET` to protect endpoints (backend guards).
- Password recovery using the Supabase flow (recovery email + reset on `/reset-password`).

### Admin and Vendor (dashboard)
- Admin pages (UI structure) for orders, products, users, vendors, and vendor applications.
- Vendor application flow (apply + status) and approval endpoints.
- Vendor order endpoints and vendor lookup associated to the current user.

### Important note about data sources
In the current state, multiple sources coexist (development-stage tradeoff):
- Part of the UI uses demo static data in `src/data/products`.
- Several API routes and repositories use PostgreSQL via `pg` (`src/lib/db.ts`).
- Some features use Supabase HTTP (`@supabase/supabase-js`) with the service role key.

The roadmap proposes standardizing this to avoid duplication and improve consistency.

## Architecture
### App Router
- UI: `src/app/**/page.tsx`
- Backend: `src/app/api/**/route.ts`

### Route protection (Next.js 16)
- Protected access is enforced via `src/proxy.ts` (Next 16 “Proxy/Middleware” convention).
- Authorization is based on the `auth_token` cookie generated by the backend.

### Data access
- Repositories live in `src/lib/repositories/*`.
- Some modules use direct SQL over Postgres (`pg` pool).
- Some modules use the Supabase client when it makes sense (HTTP) with service role.

## Main modules and routes
### Pages (UI)
- Storefront: `/`, `/shop`, `/collections`, `/products/[id]`, `/cart`, `/checkout`, `/wishlist`
- Auth: `/login`, `/register`, `/forgot-password`, `/reset-password`
- Dashboard: `/dashboard/*` (orders/products/users/etc.)
- Admin: `/admin/*`
- Vendor: `/vendor/apply`, `/vendor/status`

### API (backend)
Endpoints live in `src/app/api/**/route.ts`.

- Auth
  - `POST /api/auth/register`
  - `POST /api/auth/login`
  - `POST /api/auth/logout`
  - `GET /api/auth/me`
  - `POST /api/auth/forgot-password`
- Products
  - `GET /api/products`
  - `POST /api/products` (admin/vendor)
  - `GET/PUT/DELETE /api/products/[id]`
- Orders
  - `POST /api/orders` (authenticated user)
  - `GET /api/orders` (admin)
  - `GET /api/orders/[id]`
- Coupons
  - `POST /api/coupons/validate`
  - `POST /api/coupons/use`
- Wishlist
  - `GET/POST/DELETE /api/wishlist`
- Vendors
  - `GET /api/vendors`
  - `GET /api/vendors/me` (admin/vendor)
  - `POST /api/vendor/apply`
  - `POST /api/vendor/approve`
  - `GET /api/vendor/orders`

## Roles
In the database (enum `user_role` in `database/all.sql`):
- `admin`
- `vendor`
- `customer`
- `moderator`
- `provider`

Note: there are UI sections for `store-manager`, but the `store_manager` role is not defined in the current enum; that area is considered UI/demo until roles and guards are standardized in the DB.

## Known limitations (current state)
- Some UI views consume demo static data (`src/data/products`) while the backend already has endpoints and repositories against Postgres.
- Some dashboard/store-manager pages use mock data or placeholder logic.
- There is no real payment integration yet (only checkout flow + order persistence).

## Environment variables
For local development create `.env.local`. For Vercel, configure the same values in **Project Settings → Environment Variables**.

A complete template exists in `.env.example`.

Minimum required:
- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY` (server-only)
- `AUTH_SECRET`

Database connection (choose one option):
- `SUPABASE_DB_URL` (recommended) or `DATABASE_URL`
- or `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`

## Run locally
Requirements:
- Node.js 18+

Commands:
- `npm install`
- `npm run dev`
- `npm run lint`
- `npm run build`
- `npm start`

Open: `http://localhost:3000`

## Database setup (Supabase)
1. Create a Supabase project.
2. Run `database/all.sql` in the SQL Editor (schema `public`).
3. Configure environment variables (recommended: `SUPABASE_DB_URL`).
4. Create users from `/register` (recommended) to keep consistency between Supabase Auth and the `usuarios` table.

About seed data:
- `database/all.sql` includes sample data (roles, categories, demo vendor, coupon). Remove it if you want a clean database.

## Deploy to Vercel
1. Push the repo to GitHub.
2. Import the project in Vercel.
3. Configure environment variables (use `.env.example` as a guide).
4. Deploy.

Security notes:
- Only variables prefixed with `NEXT_PUBLIC_` are exposed to the browser.
- Never expose `SUPABASE_SERVICE_ROLE_KEY` publicly.
